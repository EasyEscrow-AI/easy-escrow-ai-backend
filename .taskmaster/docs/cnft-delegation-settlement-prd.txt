# cNFT Delegation-Based Settlement PRD
# Replacing JITO Bundles with Bubblegum Delegate Authority

## Overview

This PRD outlines the implementation of a delegation-based settlement system for compressed NFTs (cNFTs) and bulk swaps, replacing the current JITO bundle approach which is not working reliably.

The new approach uses Metaplex Bubblegum's delegation primitive, where the marketplace program PDA becomes the delegate and can drive settlement across multiple normal transactions without needing cross-tx atomicity.

## Core Principles

1. **Singles (sales, offers, 1-for-1 where it fits)**: One atomic settlement tx (already fully working)
2. **Everything "cNFT" or "bulk"**: Lock → Chunked Settle → Finalize with on-chain progress, using Bubblegum delegation

## Target Environment

- **Initial deployment**: Staging server (devnet)
- **Production deployment**: After staging validation

### Task 5: Implement cNFT Buy with Delegate Authority

**Priority**: High
**Dependencies**: Task 4

**Description**: Implement buy flow using delegate authority transfer.

**Flow**:
1. Buyer sends SOL to escrow
2. Backend builds Bubblegum Transfer CPI using delegate authority
3. Single atomic transaction:
   - SOL transfer from buyer → seller
   - cNFT transfer via delegate (buyer) ← seller's delegated cNFT
   - Platform fees paid
   - Listing closed

**Key Points**:
- Transfer is valid because delegate (marketplace PDA) signs
- No need for seller to be online for purchase
- Single transaction = atomic settlement

## Feature 4: Two-Phase cNFT Swap (Multi-Asset)

### Task 8: Design Two-Phase Swap State Machine

**Priority**: High
**Dependencies**: Task 3

**Description**: Design state machine for two-phase swaps.

**States**:
1. CREATED - Swap offer created, awaiting both parties
2. LOCKING_PHASE - Assets being delegated/locked
3. PARTY_A_LOCKED - One party has locked assets
4. FULLY_LOCKED - Both parties locked
5. SETTLING_PHASE - Transfers in progress
6. PARTIAL_SETTLE - Some transfers complete
7. COMPLETED - All transfers done
8. FAILED - Requires manual intervention
9. CANCELLED - User cancelled before lock

**Database Schema**:
- SwapState: id, status, partyA, partyB, assetsA, assetsB, lockTxA, lockTxB, settleTxs[], completedAt

---

### Task 9: Implement Lock Phase (Delegation)

**Priority**: High
**Dependencies**: Task 8

**Description**: Implement Phase A - both parties delegate assets to PDA.

**Flow**:
1. Create swap intent
2. Build delegate instructions for Party A's assets
3. Party A signs → assets delegated to marketplace PDA
4. Optionally freeze assets (Bubblegum v2)
5. Build delegate instructions for Party B's assets
6. Party B signs → assets delegated
7. Update swap state to FULLY_LOCKED

**Safety Guarantees**:
- Delegated/frozen assets cannot be moved by owner
- Swap is resumable if one tx fails
- Neither party can "rug" mid-flight

---

### Task 10: Implement Settle Phase (Chunked Transfers)

**Priority**: High
**Dependencies**: Task 9

**Description**: Implement Phase B - execute transfers in chunks.

**Flow**:
1. Once FULLY_LOCKED, begin settlement
2. Execute transfers in batches (1-2 cNFTs per tx based on proof size)
3. Track progress in SwapState.settleTxs
4. Handle partial failures with retry logic
5. Mark COMPLETED when all transfers done

**Chunking Strategy**:
- 1 cNFT per tx if proofs are large (deep trees)
- 2 cNFTs if proofs fit in tx size limit
- Progress tracked on-chain and in database

---

### Task 11: Implement Swap State Recovery and Monitoring

**Priority**: High
**Dependencies**: Task 10

**Description**: Implement recovery for interrupted swaps.

**Recovery Scenarios**:
1. Lock tx failed → Retry lock, revert if persistent
2. One party locked, other didn't → Timeout and revoke delegation
3. Partial settlement → Resume from last confirmed tx
4. Complete failure → Admin intervention, refund delegated assets

**Monitoring**:
- Dashboard for swap states
- Alerts for stuck swaps
- Automatic retry for transient failures
