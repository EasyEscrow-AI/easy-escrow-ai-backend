# Task ID: 45
# Title: Implement Jito Bundle Submission for Bulk Swap Parallel Execution
# Status: pending
# Dependencies: 41
# Priority: high
# Description: Extend the existing Jito integration to support bundle submission for bulk swaps with 3+ total cNFTs, enabling atomic parallel transaction execution with proper ordering, simulation, and confirmation handling.
# Details:
Implement comprehensive Jito bundle submission functionality in src/services/escrow-program.service.ts with the following components:

1. **Extend Existing Jito Integration**:
   - Build upon existing sendTransactionViaJito() method documented in docs/JITO_FREE_INTEGRATION_SUMMARY.md
   - Maintain compatibility with current single transaction Jito submissions
   - Reuse existing Jito endpoint configuration and authentication setup

2. **Bundle Creation and Management**:
   - Implement sendBundleViaJito() method that accepts array of Transaction objects
   - Create Jito bundle with proper transaction ordering (nonce advance first, transfers, fee collection last)
   - Implement bundle size validation (max 5 transactions per bundle as per Jito limits)
   - Add bundle threshold logic: use bundles for swaps with 3+ total cNFTs (since 1-2 cNFTs fit in single transaction, but 3+ require multiple transactions due to account limits), otherwise use single transaction

3. **Dynamic Tip Calculation**:
   - Implement dynamic Jito tip calculation based on current network congestion
   - Query Jito tip stream API for recommended tip amounts
   - Include tip as final instruction in last transaction of bundle
   - Add fallback tip amounts for API failures (0.001 SOL default)

4. **Bundle Simulation and Validation**:
   - Implement bundle simulation using Jito's simulateBundle endpoint before submission
   - Validate all transactions in bundle pass simulation
   - Handle simulation failures with detailed error reporting
   - Add retry logic with fresh Merkle proofs on simulation failures

5. **Bundle Submission and Confirmation**:
   - Submit bundle via Jito's sendBundle endpoint
   - Implement bundle ID tracking in database with new bundle_id column in swap_transactions table
   - Use getBundleStatuses endpoint for confirmation polling
   - Handle bundle landing rate monitoring and timeout scenarios (30 second timeout)

6. **Error Handling and Retry Logic**:
   - Implement exponential backoff retry for bundle failures
   - Refresh Merkle proofs on retry attempts for cNFT transactions
   - Handle partial bundle confirmation scenarios
   - Add fallback to individual transaction submission on repeated bundle failures

7. **Comprehensive Logging**:
   - Log complete bundle lifecycle: creation, simulation, submission, confirmation
   - Include bundle ID, transaction count, tip amount, and timing metrics
   - Add structured logging for bundle performance analytics
   - Implement error categorization for bundle failure analysis

8. **Integration with Existing Services**:
   - Update TransactionBuilder service to flag transactions suitable for bundling
   - Modify OfferManager to trigger bundle submission for qualifying bulk swaps
   - Ensure compatibility with existing nonce pool management and fee calculation
