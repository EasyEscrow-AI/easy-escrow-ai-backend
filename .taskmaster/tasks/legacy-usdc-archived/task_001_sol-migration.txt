# Title: Replace USDC with SOL in Solana Escrow Program and Implement SOL-Based Swap Types
# Status: pending
# Dependencies: None
# Priority: high
# Description: Refactor the Solana escrow smart contract and all related integrations to remove USDC and support SOL-based swaps for NFT<>SOL, NFT<>NFT (with SOL fee), and NFT<>NFT+SOL (fee from SOL). This is the foundational task that enables SOL-based escrow functionality across the platform.
# Details:
1. **Smart Contract Refactor:** Remove all USDC token references and SPL token handling related to USDC. Implement native SOL handling for escrow swaps using Solana's system program for SOL transfers. 

2. **Swap Types Implementation:**
  - NFT <> SOL: Enable direct escrow swaps where NFTs are exchanged for SOL.
  - NFT <> NFT with SOL fee: Support NFT-for-NFT swaps where the buyer pays a commission fee in SOL.
  - NFT <> NFT+SOL: Support swaps where the buyer pays an additional SOL fee deducted from the SOL amount.

3. **Backend Integration:** Update backend logic to interact with the modified escrow program, handling SOL transfers and NFT escrow states accordingly.

4. **API Endpoints:** Modify API endpoints to reflect the new swap types and fee structures, ensuring clear parameters for SOL amounts and fee payer defaults.

5. **Frontend Integration:** Update UI components to support SOL-based swaps, display fees in SOL, and handle user wallet interactions for SOL payments.

6. **Documentation:** Revise all technical and user documentation to remove USDC references and explain the new SOL-based swap mechanisms and fee policies.

7. **Phased Deployment:** Implement the changes in development, then staging, and finally production environments, ensuring stability and correctness at each phase.

8. **Best Practices:** Use Anchor framework for smart contract development to leverage security and testing features. Follow Solana's native SOL transfer patterns and ensure proper handling of lamports. Implement comprehensive error handling and state validation.

# Test Strategy:
- Develop comprehensive end-to-end tests covering all three swap types under happy path scenarios:
  * NFT <> SOL swap: Verify escrow creation, SOL transfer, NFT transfer, and escrow closure.
  * NFT <> NFT swap with SOL fee: Verify NFT exchanges and SOL fee payment by buyer.
  * NFT <> NFT+SOL swap: Verify combined NFT and SOL transfer with fee deduction.
- Test default commission payer behavior (buyer pays commission).
- Perform integration tests for backend and API endpoints to validate correct data flow and SOL handling.
- Conduct frontend UI tests to ensure correct user interactions and SOL fee displays.
- Execute phased deployment testing: full test suite run and pass in development, then staging, before production rollout.
- Include negative and edge case tests to ensure robustness against invalid inputs or failed transfers.

<info added on 2025-11-04T00:52:26.938Z>
SOL-based architecture design completed with comprehensive documentation in `docs/architecture/SOL_MIGRATION_ARCHITECTURE.md`. The architecture supports three distinct swap types: NftForSol (direct NFT-SOL exchange), NftForNftWithFee (NFT-NFT with separate SOL fee), and NftForNftPlusSol (NFT-NFT+SOL with fee extracted from SOL amount). 

New state structure includes SwapType enum, sol_amount field replacing usdc_amount, optional nft_b_mint for buyer's NFT, deposit tracking flags, and FeePayer enum. Seven new instructions designed: init_agreement_v2, deposit_sol, deposit_buyer_nft, deposit_seller_nft, settle_v2, cancel_if_expired_v2, and admin_cancel_v2.

SOL transfers utilize Solana's system program for direct lamports transfers to/from PDA, providing ~49,000 compute units savings versus USDC. Beta limits set at 0.01-15 SOL range. Feature flag strategy implemented with USDC code behind conditional compilation, ensuring clean separation for legal compliance while maintaining internal ATA utilities.

Security features include locked fee BPS at initialization, admin-only escrow creation, atomic settlement, proper PDA signer seeds, and rent-exempt checks. Architecture is complete and implementation-ready.
