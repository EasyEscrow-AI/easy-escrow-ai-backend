# Details:
Develop comprehensive documentation in docs/architecture/STAGING_STRATEGY.md covering the following key areas:

**1. STAGING Environment Overview**
- Purpose: Production-like testing on devnet (release candidate gate)
- Not a manual deploy environment - CI/CD only
- Promotion path: Local → DEV → STAGING → PROD
- Build once, promote everywhere (same .so and IDL)

**2. Why Devnet for STAGING (Not Testnet)**
- Devnet: Public testing playground, faucet access, stable for app testing
- Testnet: Validator/core stress testing, newer branches, can be flaky
- Mainnet-shadow: Small gated real mainnet transactions (separate step)

**3. Environment Separation Strategy**
Even though DEV and STAGING both use devnet:
- Separate Program IDs: DEV can be noisy; STAGING clean and stable
- Separate RPC endpoints: Private RPC per env to isolate rate limits
- Separate infrastructure: DB, Redis, queues, indexers, monitoring
- Separate wallets: Distinct funder/test wallets, independent balance management
- Separate configs: Anchor.dev.toml vs Anchor.staging.toml

**4. Build-Once-Promote Pattern**
Merge to staging branch → CI: Build (pin Solana 1.18.x, Rust 1.75.0) → Generate artifacts (.so, IDL, checksums) → Unit tests (no public chain) → Store artifacts → [Manual Approval Gate] → CI: Promote to STAGING → Deploy same .so and IDL → Smoke tests → E2E tests → Mark as RC for PROD

**5. CI/CD Requirements**
- Never deploy from laptop ("ssh + deploy" is forbidden)
- All deploys through GitHub Actions or similar CI
- Pinned toolchains (reproducible builds)
- Artifact checksums for verification
- Auditable (who/what/when logged)
- Automatic rollback capability

**6. One-Time Setup**
- Separate Program IDs per env (DEV, STAGING, PROD)
- Keep keypairs encrypted in repo or secure store (never reuse IDs)
- Anchor.dev.toml and Anchor.staging.toml configs
- CI secrets: deployer keypairs, RPC URLs, seed mints
- Use ANCHOR_CONFIG env var or -C flag to switch configs

**7. Upgrade Authority Management**
- Keep upgrade authority in safe signer (multisig, not laptop)
- Can transfer authority later or make program immutable
- Document who holds upgrade authority
- Plan for emergency upgrades

**9. Canary Deployments and Rollback**
After any deployment:
- Run synthetic canary swaps with tiny amounts
- Monitor metrics and error rates
- Have rollback plan ready (< 5 minutes)
- Keep previous deployment artifacts

**10. What STAGING Provides**
✅ Stable RC lane separate from DEV churn
✅ Prod-like topology (API, queues, websockets, DB, metrics)
✅ Repeatable E2E before mainnet - no real funds at risk
✅ Performance baseline establishment
✅ Security testing gate
✅ Confidence for production promotion

**11. Migration from DEV to STAGING**
- Promote code, NOT keys
- Each environment has distinct credentials
- Test database migrations on STAGING first
- Verify backward compatibility
- Run full test suite before marking RC ready

**12. Mainnet-Shadow Testing (Separate from STAGING)**
- Small, gated real mainnet transactions
- Validates production conditions devnet can't reproduce
- Not part of STAGING - separate testing phase
- Will be implemented before PROD launch
