## 1. Design Idempotency Key Schema and Storage [done]
### Dependencies: None
### Description: Design database schema for storing idempotency keys with metadata including endpoint, request hash, response data, and expiration timestamps
### Details:
Create database table with fields: idempotency_key (unique), endpoint_path, request_hash, response_status, response_body, created_at, expires_at. Define TTL for key expiration (24-48 hours). Design indexes for efficient lookups.

## 2. Implement Idempotency Key Validation Middleware [done]
### Dependencies: 40.1
### Description: Create middleware to extract, validate, and process idempotency keys from request headers for critical endpoints
### Details:
Extract Idempotency-Key header, validate format (UUID v4), check for required endpoints (/v1/agreements, /v1/agreements/{id}/settle). Return 400 for missing/invalid keys on protected endpoints.

## 3. Implement Duplicate Request Detection Logic [done]
### Dependencies: 40.1, 40.2
### Description: Build service to detect duplicate requests by comparing idempotency keys and request content hashes
### Details:
Generate SHA-256 hash of request body and parameters. Check database for existing idempotency key. If found with same hash, return cached response. If found with different hash, return 409 Conflict.

## 4. Integrate Idempotency with Create Agreement Endpoint [done]
### Dependencies: 40.2, 40.3
### Description: Add idempotency key processing to the create agreement endpoint to prevent duplicate agreement creation
### Details:
Wrap agreement creation logic with idempotency checks. Store successful responses with 201 status. Handle race conditions with database constraints. Ensure atomic operations between agreement creation and idempotency key storage.

## 5. Integrate Idempotency with Settlement Endpoint [done]
### Dependencies: 40.2, 40.3
### Description: Add idempotency key processing to the settlement endpoint to prevent double-settlement of agreements
