# Environment-Specific IDL Management

## Overview

This project uses **environment-specific IDL (Interface Definition Language) files** to ensure the backend always communicates with the correct Solana program deployment. Each environment (dev, staging, production) has its own program deployment with a unique program ID, and therefore requires its own IDL file.

## Why Separate IDL Files?

### The Problem
- Each Anchor program deployment has a **unique program ID**
- The program ID is embedded in the **IDL's `address` field**
- The backend uses this IDL to interact with the on-chain program
- Using the wrong IDL causes **program ID mismatch errors**

### The Solution
- Maintain **separate IDL files for each environment**
- Dynamically **load the correct IDL** based on `NODE_ENV`
- Automate IDL generation and distribution during builds/deployments


## How It Works

### 1. IDL Generation

IDLs are generated using the `copy-idl-for-env.ps1` script:

```powershell
# Generate dev IDL
.\scripts\utilities\copy-idl-for-env.ps1 -Environment dev -ProgramId 4FQ5JoxsS5jjuTR1ScuEpk66eX5B71L7ysJEysmsTwhd

# Generate staging IDL  
.\scripts\utilities\copy-idl-for-env.ps1 -Environment staging -ProgramId AvdX6LEkoAmP961QwNjAUNpiuDtiQjaiSw5wR5zb9Zei

# Generate all at once (if all program IDs are known)
.\scripts\utilities\build-all-idls.ps1
```

### 2. IDL Distribution

Copy environment-specific IDLs to the backend source:

```powershell
.\scripts\utilities\copy-env-idls.ps1
```

This copies from `target/idl/escrow-*.json` to `src/generated/anchor/escrow-idl-*.json`.

### 3. Dynamic Loading

The backend uses `src/utils/idl-loader.ts` to load the correct IDL:

```typescript
import { getEscrowIdl } from '../utils/idl-loader';

// Automatically loads the correct IDL based on NODE_ENV
const escrowIdl = getEscrowIdl();
const program = new Program<Escrow>(escrowIdl, provider);
```

**Environment mapping:**
- `NODE_ENV=development` or `NODE_ENV=dev` → loads `escrow-idl-dev.json`
- `NODE_ENV=staging` → loads `escrow-idl-staging.json`
- `NODE_ENV=production` → loads `escrow-idl-production.json` (when available)

## Workflows

### Creating a New Environment IDL

When you deploy a new program to an environment:

1. **Copy and update the IDL:**
   ```powershell
   .\scripts\utilities\copy-idl-for-env.ps1 -Environment staging -ProgramId <YOUR_PROGRAM_ID>
   ```

2. **Copy to backend source:**
   ```powershell
   .\scripts\utilities\copy-env-idls.ps1
   ```

3. **Build the backend:**
   ```bash
   npm run build
   ```

4. **Verify:**
   ```bash
   # Set environment
   $env:NODE_ENV="staging"
   
   # Start backend and check logs for:
   # [IDL Loader] Loaded staging IDL with program ID: AvdX6LEkoAmP961QwNjAUNpiuDtiQjaiSw5wR5zb9Zei
   npm start
   ```

### Updating an Existing Environment IDL

If you redeploy a program to an existing environment with a new program ID:

1. **Update the environment-specific IDL:**
   ```powershell
   .\scripts\utilities\copy-idl-for-env.ps1 -Environment staging -ProgramId <NEW_PROGRAM_ID>
   ```

2. **Copy to backend:**
   ```powershell
   .\scripts\utilities\copy-env-idls.ps1
   ```

3. **Rebuild and redeploy backend:**
   ```bash
   npm run build
   npm run staging:deploy
   ```

### Deploying to a New Environment

1. **Deploy the Anchor program** to the new environment
2. **Note the program ID** from the deployment receipt
3. **Generate the IDL** using the script
4. **Update the environment variables** (e.g., `ESCROW_PROGRAM_ID`, `SOLANA_RPC_URL`)
5. **Deploy the backend** with the new IDL included in the build

## Metadata Files

Each environment IDL has an accompanying metadata file (e.g., `escrow-dev.metadata.json`):

```json
{
  "environment": "staging",
  "programId": "AvdX6LEkoAmP961QwNjAUNpiuDtiQjaiSw5wR5zb9Zei",
  "generatedAt": "2025-10-22T11:41:58",
  "originalProgramId": "4FQ5JoxsS5jjuTR1ScuEpk66eX5B71L7ysJEysmsTwhd"
}
```

**Purpose:**
- Track which environment each IDL belongs to
- Record when the IDL was generated
- Document the source program ID (for debugging)

## Integration with Build Process

### Development

```bash
# Normal development - uses dev IDL
npm run dev
```

### Staging Deployment

```bash
# Build with staging IDL
$env:NODE_ENV="staging"
npm run build

# Deploy
npm run staging:deploy
```

### CI/CD Pipeline

In your CI/CD pipeline, ensure:

1. **Generate IDLs** during the build step:
   ```yaml
   - name: Generate environment-specific IDLs
     run: |
       .\scripts\utilities\copy-idl-for-env.ps1 -Environment ${{ env.ENVIRONMENT }} -ProgramId ${{ secrets.PROGRAM_ID }}
       .\scripts\utilities\copy-env-idls.ps1
   ```

2. **Include IDLs in the build artifact:**
   ```yaml
   - name: Build backend
     run: npm run build
   ```

3. **Set NODE_ENV during deployment:**
   ```yaml
   - name: Deploy
     env:
       NODE_ENV: staging
     run: npm run staging:deploy
   ```

## Troubleshooting

### Error: Program ID mismatch

```
[EscrowProgramService] Program ID mismatch: 
  IDL has 4FQ5JoxsS5jjuTR1ScuEpk66eX5B71L7ysJEysmsTwhd, 
  config has AvdX6LEkoAmP961QwNjAUNpiuDtiQjaiSw5wR5zb9Zei
```

**Cause:** The backend is loading the wrong IDL for the current environment.

**Solution:**
1. Verify `NODE_ENV` is set correctly
2. Check that environment-specific IDL exists: `src/generated/anchor/escrow-idl-staging.json`
3. Regenerate and copy IDLs:
   ```powershell
   .\scripts\utilities\copy-idl-for-env.ps1 -Environment staging -ProgramId AvdX6LEkoAmP961QwNjAUNpiuDtiQjaiSw5wR5zb9Zei
   .\scripts\utilities\copy-env-idls.ps1
   npm run build
   ```

### Error: Cannot find module 'escrow-idl-staging.json'

**Cause:** Environment-specific IDL file is missing.

**Solution:**
1. Generate the IDL:
   ```powershell
   .\scripts\utilities\copy-idl-for-env.ps1 -Environment staging -ProgramId <PROGRAM_ID>
   ```
2. Copy to source:
   ```powershell
   .\scripts\utilities\copy-env-idls.ps1
   ```

### Wrong IDL loaded despite correct NODE_ENV

**Cause:** Build artifacts contain old IDL files.

**Solution:**
```bash
# Clean build
rm -rf dist/
npm run build
```

## Best Practices

1. **Always regenerate IDLs** after deploying a new program version
2. **Commit IDL files** to version control (they're essential for deployment)
3. **Document program IDs** in this file and in environment-specific documentation
4. **Verify IDL matches** before deploying to production
5. **Use metadata files** to track IDL provenance
6. **Test locally** with the correct `NODE_ENV` before deploying
